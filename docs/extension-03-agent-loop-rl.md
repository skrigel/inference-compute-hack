# Extension 03: Agentic Iteration Loop And Prime Training Readiness

This extension turns the existing query/refine loop into a small RL-style
environment and a pre-Prime training handoff. The goal is to prove the applied
AI claim without bloating the demo: cheap inference lets an agent run many fast
query-refinement attempts, verify evidence, and stop when the result crosses a
quality threshold.

The current implementation is intentionally split in two layers:

1. `eval.agent_loop`: executable dynamic-corpus environment and cheap metrics.
2. `eval.agent_loop_prime`: cohort manifests, reward contract, training config
   template, metric-to-lift schema, and no-credit readiness smoke.

## Thesis

Humans underuse cheap iteration because every failed query costs attention and
time. An agent can spend cheap inference to repeatedly refine a query, inspect
scored evidence, and stop when reward crosses a target. This is the use case
where abundant inference matters: not one better query, but many fast
self-corrections.

The Applied AI track question asks which task-level and dataset-level metrics
correlate with post-RL performance gain. Our proposed answer is:

- Task-level metrics such as reward variance, steps-to-threshold, branching
  factor, byte movement, and truth gain measure how much useful learning signal
  a rollout exposes before training.
- Dataset-level metrics such as trajectory entropy, task diversity, memory
  selectivity, movement selectivity, and pass rate summarize whether a cohort is
  likely to create learnable policy improvements rather than redundant examples.
- The cost-quality frontier is metric cost versus predictive power for heldout
  post-RL lift.

## Infinite-Compute 3-Axis Mapping

| axis | environment metric | rationale |
|---|---|---|
| memory capacity used | `memory_selectivity` | fraction of stored bytes that are actually positive evidence |
| movement / bandwidth | `movement_selectivity`, `bytes_moved_total` | bytes moved/scored/selected during rollouts |
| answer error | `precision`, `recall`, `F1`, `truth_gain` | whether the final selected evidence supports the task |

Rule of thumb: spend extra compute on query planning and verification, not on
blindly storing or moving more data.

## Applied AI Track Primitives

| primitive | project definition |
|---|---|
| `T` | query-refinement task over a labeled synthetic dynamic corpus |
| `M` | policy/model that proposes the next query or stop action |
| `V` | verifier comparing selected chunks to positive chunk ids |
| `y` | candidate refined query and selected evidence set |
| `r` | `F1 + recall bonus - byte movement/storage penalties` |

In the smoke implementation, `M` is represented by deterministic candidate
query expansion plus the mock scorer. In the training version, `M` becomes the
policy model being post-trained.

## Cohort Design

Each train cohort varies one property at a time against `c00_baseline_retry`.
Heldout cohorts are separated so metric-to-lift claims are not fit and judged on
the same examples.

| cohort | split | changed variable | expected signal |
|---|---|---|---|
| `c00_baseline_retry` | train | baseline | control cohort for retry/backoff tasks |
| `c01_topic_ir` | train | `target_topic` | tests transfer to retrieval/ranking language |
| `c02_topic_cache` | train | `target_topic` | tests transfer to cache/threshold language |
| `c03_sparse_positive` | train | `positive_ratio` low | harder recall, stronger verifier signal |
| `c04_dense_positive` | train | `positive_ratio` high | easier recall, lower selectivity pressure |
| `c05_hard_distractors` | train | `distractor_hardness` | tests precision under near-miss negatives |
| `c06_large_dynamic` | train | `n_docs` | tests scaling under generated-on-the-spot corpora |
| `c07_long_rollout` | train | `max_steps` | tests value of deeper agent iteration |
| `c08_reward_noise` | train | `reward_noise` | tests robustness to noisy RLAIF-style feedback |
| `h00_heldout_retry` | heldout | split | heldout retry/backoff eval |
| `h01_heldout_ir` | heldout | split | heldout retrieval eval |
| `h02_heldout_cache` | heldout | split | heldout cache eval |

Primary predictors to compute before training:

- `mean_reward_variance`
- `trajectory_entropy`
- `mean_steps_to_threshold`
- `mean_memory_selectivity`
- `mean_movement_selectivity`
- `agent_vs_human_speedup_estimate`

## Reward Contract

The committed reward contract uses:

```text
reward = F1 + 0.25 * recall - 0.05 * movement_ratio - 0.10 * selected_ratio
```

Where:

- `movement_ratio = bytes_moved_this_rollout / total_corpus_bytes`
- `selected_ratio = bytes_selected_final / total_corpus_bytes`
- pass gate is `recall >= 0.8 and precision >= 0.5`

The reward is deliberately hostile to the trivial "select every chunk" strategy:
select-all can improve recall, but it pays selected-byte penalties and loses
precision. The training harness must log these failure modes:

- `query_overbroad_high_recall_low_precision`
- `query_underspecified_low_recall`
- `early_stop_before_pass_gate`
- `byte_budget_exceeded`
- `reward_hacking_by_selecting_all_chunks`

## Training Protocol

The pilot protocol is one adapter run per cohort with all non-cohort settings
held fixed.

1. Freeze this repo commit, cohort manifest, reward contract, base model, and
   Prime config.
2. Compute cheap pre-training metrics for every train and heldout cohort.
3. Run baseline eval of the base model on heldout cohorts.
4. For each train cohort, launch one identical GRPO/post-training job.
5. Evaluate each trained adapter on the same heldout cohorts.
6. Compute lift: `trained_reward - baseline_reward`.
7. Fit cheap metrics to lift and report `R2`, RMSE, and Spearman rank.
8. Plot the cost-quality Pareto frontier: metric cost proxy versus predictive
   power.

Candidate pilot model in the config is `Qwen/Qwen2.5-1.5B-Instruct` with LoRA.
This is a conservative pilot choice; verify current Prime image/model support
and pricing before launch. Move to a larger model only after the pipeline has a
passing tiny run.

## Prime Launch Gate

The paid command is:

```bash
prime train run eval/configs/extension3_prime/prime_train.example.toml
```

Do not run it until every item is true:

- local no-credit readiness report passed;
- Modal CPU smoke passed;
- Prime CLI is installed and authenticated;
- the Verifiers environment package has been created from the reward contract;
- baseline eval artifact exists for heldout cohorts;
- cohort manifest has at least one train and one heldout cohort;
- user explicitly approves spending Prime credits for this exact run.

Stop after the first paid pilot if:

- reward becomes NaN or constant;
- heldout reward drops below baseline;
- selected bytes trend toward selecting the full corpus;
- logs do not include cohort id, run id, base model, adapter artifact, and git
  commit;
- the first adapter cannot be evaluated against heldout cohorts.

## Metric-To-Lift Output

Every post-training cohort row must conform to
`eval/configs/extension3_prime/metric_to_lift_schema.example.json`.

Required columns:

| group | fields |
|---|---|
| cohort identity | `cohort_id`, `split`, `single_changed_variable` |
| cheap metrics | `mean_best_reward`, `mean_reward_variance`, `trajectory_entropy`, `mean_steps_to_threshold`, `mean_memory_selectivity`, `mean_movement_selectivity`, `cost_proxy_model_calls` |
| training | `base_model`, `algorithm`, `adapter_artifact`, `train_wall_clock_minutes`, `estimated_cost_usd` |
| eval | `baseline_reward`, `trained_reward`, `lift`, `heldout_pass_rate` |

The final result should include:

- scatter plot: each cheap metric versus lift;
- table: `R2`, RMSE, and Spearman for every metric;
- Pareto chart: metric cost proxy versus predictive power;
- short rationale for why the strongest metric should correlate with RL lift.
